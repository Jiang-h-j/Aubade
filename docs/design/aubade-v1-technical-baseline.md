# Aubade v1 技术基线

## 给用户看的摘要

这份文档把已评审通过的 PRD 和原型，翻译成"用什么技术、怎么搭骨架、按什么顺序做、哪里有风险"的技术事实来源。它只定架构与策略，不写具体代码；每个功能的文件级实现计划留到后续节点 TRD。

**关键技术建议（含你已拍板的两条）**：

1. **平台与数据层**：iOS 17+，用 **SwiftUI + SwiftData + @Observable**。这是苹果当前最简洁的原生组合，账本十年也就几 MB，纯本地存储绰绰有余，不需要后端或 iCloud。
2. **截图主入口（你已选"后台全流程入账"）**：用 **App Intents** 暴露一个"记录截图"动作给 iOS 快捷指令。你在付款页触发后，App 在后台一口气跑完 **本地 OCR（Vision）→ DeepSeek 解析 → 直接入账 → 本地通知**，全程不打断你。
3. **识别能力分工**：图片文字识别用 iPhone 本地 **Vision**、语音转文字用本地 **Speech**，**图片和语音都不出机**；只有识别出来的**文本**才发给 **DeepSeek**（走它的 OpenAI 兼容接口）做结构化解析和分类。
4. **密钥与隐私**：DeepSeek API Key 存 **Keychain**（不进数据库、不进代码）；除了把文本发给 DeepSeek，数据不出本机。

**主要风险（都已给退路，不会让你丢数据）**：

- **后台时间预算风险**：App Intent 在后台执行有系统时间限制，一次 DeepSeek 网络往返（含慢网/超时）**有可能跑不完**。对策已定：一旦超时/失败/无网/无 Key，**不生成账单**，改为保留原图 + 发一条"没识别出，点此补录"的通知，你进 App 手动补——绝不误记脏账。**若真机实测发现后台预算普遍不够，则整条截图链路降级为"后台只做 OCR 存文本 + 通知，你点通知进 App 由前台完成解析入账"**（即当初的备选方案 B 作为兜底）。这一点必须在截图节点用**真机**验证，模拟器测不准。
- **解析质量风险**：自动记账/自动分类准不准，取决于 DeepSeek 的解析效果；任何识别结果你都能当场或事后在列表里改。
- **真机依赖**：快捷指令、App Intents、本地通知、后台联网这几项，模拟器支持不完整，**必须真机 + Xcode 调试验证**。

**需要你判断的取舍**：地基两条（iOS 17+ / 截图后台全流程）已确认。其余保留在文末"待确认"，都不阻塞开始动工，可在对应节点定。

## 2. 当前对齐口径

以下口径来自已通过评审的 PRD 与原型，作为技术设计的既定前提：

- **单用户、私人自用、不上架 App Store**，Mac + Xcode 开发，真机安装自用。
- **四种记账入口**：截图识别（快捷指令后台入账为主 + App 内相册选图为备选）、语音、文本识别（粘贴短信/任意含金额文字）、手动。
- **识别/填写完直接入账**，无待确认队列；错了在账单列表改。
- **剩余金额 = 所有账户合计总余额**：录一次初始总额，之后收入加、支出减，滚动显示；无独立多账户体系，卡尾号仅作记录字段。
- **分类体系（预置，可增删改）**：支出 衣/食/住/行/玩/其他；收入 工作/其他收入。
- **预算**：周预算 + 月预算并存；周按自然周（周一起）、月按自然月；接近阈值（默认 80%）提示、超 100% 标红，仅提示不阻止。
- **统计时间维度**：唯一一套「日/周/月/年」粒度 + 「‹ 具体时间 ›」导航，趋势横轴跟随粒度，不能翻到未来；日档列当天流水、周/月/年档给趋势+分类占比、周/月档另有预算进度。
- **隐私边界**：Vision/Speech 本地完成，图片/语音不外传；只有文本发 DeepSeek；DeepSeek Key 用户在 App 内填写、本机保存。
- **短信来源聚焦工商银行（95588）**，但文本识别入口接受任意含金额文字。
- **v1 明确不做**：闹钟、短信自动触发唤起、支付宝/微信记账、待确认队列、多账户独立体系、账单 CSV/xlsx 导入、iCloud/云端/多设备同步、数据/报表导出、深度图表分析、多币种、附件长期图库管理。

## 3. 已确认技术决策

| 领域 | 决策 | 理由 / 备注 |
|---|---|---|
| 最低系统版本 | **iOS 17+** | 用户已确认；解锁 SwiftData 与 @Observable，代码最简。 |
| UI 框架 | **SwiftUI** | 原生、私人自用、无历史包袱。 |
| 架构模式 | **SwiftUI + @Observable ViewModel（轻量 MVVM）** | 视图薄、逻辑集中在可观察的 ViewModel/Store；避免过度分层。 |
| 数据持久化 | **SwiftData（本地）** | 十年几 MB，纯本地；金额用 `Decimal` 避免浮点误差。 |
| 云 / 后端 | **无**（不接后端、不接 iCloud） | PRD 明确排除。 |
| 截图主入口 | **App Intents 后台全流程入账**（用户已确认） | 快捷指令触发 → 后台 OCR→DeepSeek→入账→通知，全程不进前台。含超时/失败兜底与真机验证要求（见 §7）。 |
| 图片文字识别 | **iOS Vision**（本地 OCR） | 图片不外传。 |
| 语音转文字 | **iOS Speech**（本地，须开启 on-device 识别） | 语音不外传；须置 `requiresOnDeviceRecognition = true` 并检查 `supportsOnDeviceRecognition`，中文本地识别不可用时按 §11 降级。 |
| 文本智能解析 + 分类 | **DeepSeek**，走其 **OpenAI 兼容 HTTP 接口**，仅传文本 | 单次结构化解析调用（非多轮 agent）；要求返回结构化 JSON（金额/方向/时间/商户/卡尾号/分类）。 |
| 密钥存储 | **Keychain** | Key 不进数据库、不进源码、不进 UserDefaults。 |
| 本地通知 | **UNUserNotificationCenter** | 后台入账结果/失败以本地通知呈现，点击带 transaction 标识跳转。 |
| 轻量设置项 | **UserDefaults**（预算周期规则、通知开关等非敏感项） | 敏感项走 Keychain，业务实体走 SwiftData。 |
| 网络层 | **URLSession**（原生，无第三方依赖） | 仅 DeepSeek 一个外部端点。 |
| 三方依赖 | **v1 默认零三方依赖** | 降低维护面；如个别处确需再在节点 TRD 单独论证。 |

## 4. 模块拆解与待办需求

按职责分层，每层列出对应 PRD/原型的待办范围。节点 DAG 将据此切分可开发节点。

### M1 数据层（Persistence Core）
- SwiftData 模型：`Transaction`、`Category`、`Budget`、`BalanceBaseline`（详见 §8）。
- `ModelContainer` 初始化与共享（含 App Intent 后台进程如何访问同一存储，见 §7/§11）。
- 预置分类的首次装载（衣/食/住/行/玩/其他 + 工作/其他收入）。
- 基础读写封装（供 ViewModel 调用的查询/增删改）。

### M2 记账入口层（Capture）
- **M2.1 手动记账**：金额/方向/分类/日期/备注表单 → 保存入账。
- **M2.2 文本识别**：粘贴框 + 读取剪贴板 → 调用 M4 解析 → 结果卡片 → 直接入账。
- **M2.3 语音记账**：Speech 转文字 → 调用 M4 解析 → 结果卡片 → 入账。
- **M2.4 截图·相册选图（备选）**：相册取图 → Vision OCR → M4 解析 → 结果卡片 → 入账（前台流程）。
- **M2.5 截图·快捷指令后台入账（主入口）**：App Intent 接收图片 → 后台 Vision OCR → M4 解析 → 直接入账 → 本地通知（含失败兜底）。

### M3 结果卡片与账单编辑（Review & Edit）
- 识别结果卡片（金额/方向/分类/时间/商户/备注 + 折叠原文），"完成"保留修改、"删除这笔"撤销入账（二次确认）。
- 账单列表（按日期分组、分类彩色标签、金额醒目）、分类/时间范围筛选、详情编辑、删除（二次确认）。

### M4 AI 解析层（DeepSeek Client）
- 组装 prompt（原始文本 + 当前分类清单 + 输出 JSON 约束）。
- 调用 DeepSeek，解析返回：金额、收支方向、时间、商户、卡尾号（若有）、推断分类。
- 金额/时间归一（元；识别不到时间取当前时间）、分类兜底（不匹配则归"其他/其他收入"）。
- 失败/超时/无 Key 的错误类型定义（供入口层决定通知或转手动）。

### M5 统计与聚合层（Analytics）
- 按「日/周/月/年」区间聚合支出/收入合计。
- 分类占比（当前区间内按分类汇总）与明细下钻。
- 趋势序列（横轴跟随粒度：周/日档=本周每日、月档=当月每日、年档=当年每月）。
- 预算进度计算（周/月档）与超支状态（≥80% 接近、>100% 超支）。

### M6 剩余金额（Balance）
- 初始总额录入/调整（`BalanceBaseline`）。
- 剩余 = 初始总额 + 基线时间点后的收入 − 之后的支出；仅在有基线时展示，否则显示"—"。

### M7 通知层（Notifications）
- 后台入账成功/失败的本地通知构造与发送（成功含金额+分类+商户+"查看/修改"；失败含"点此补录"）。
- 点击通知的路由（成功→结果卡片；失败→手动补录并带出原文/原图）。
- 通知权限申请与设置里的开关。

### M8 设置与安全（Settings & Security）
- DeepSeek API Key 的 Keychain 读写与"已配置/去填写"状态。
- 预算设置（周/月）、通知开关、分类查看/增删改。
- 首次启动引导（录初始总额 → 提示配置 Key，可跳过）。

### M9 平台集成与容错（Platform & Resilience）
- 快捷指令接入形态与设置指引文案。
- 原图临时留存与清理（成功入账或放弃后清理；失败保留供补录）。
- 权限（相册、麦克风+语音识别、通知）申请与被拒降级。
- 未配置 Key 的统一拦截提示（前台拦截 / 后台以通知提示）。

## 5. 建议开发顺序

按"先打通不依赖 AI 的最短闭环，再逐个接入识别能力，最复杂的后台链路放最后"排序：

1. **M1 数据层地基** → 模型、容器、预置分类、基线。
2. **M2.1 手动记账 + M3 账单列表/编辑** → 打通"记一笔 → 看列表 → 改/删"的完整闭环，完全不依赖 AI。
3. **M6 剩余金额 + M5 统计基础** → 依赖账单数据，先出合计与剩余，再补趋势/占比/预算。
4. **M4 DeepSeek Client + M2.2 文本识别** → 第一个 AI 入口，纯文本、最易测，用来验证解析与结果卡片。
5. **M2.3 语音记账** → 复用 M4，前面加 Speech。
6. **M2.4 截图·相册选图** → 复用 M4，前面加 Vision，仍是前台流程。
7. **M2.5 截图·快捷指令后台入账 + M7 通知** → 最复杂、最依赖真机，放最后；此处验证后台时间预算与兜底降级。
8. **M8 预算/设置/Key/首次引导 + M9 权限与容错** → 贯穿收尾，补齐提示与异常路径。

> 顺序原则：每一步都能独立跑通并验收；AI 与平台能力从"最可控"到"最不可控"逐步接入，把后台链路的不确定性隔离在最后一个节点。

## 6. MVP 第一阶段边界

**第一阶段（可自用的最小闭环）= 步骤 1–3**：

- 数据层 + 手动记账 + 账单列表/编辑 + 剩余金额 + 统计合计（支出/收入 + 基础分类占比）。
- 达成后你已经能"手动记账、看流水、改删、看本周/本月花了多少、看还剩多少"，是一个不依赖任何网络/AI 就能日常用的记账 App。

**第二阶段**：接入 M4 + 文本识别（步骤 4），让"粘贴短信一键记账"跑通——这是 PRD 最核心的降本入口。

**第三阶段**：语音、相册截图、快捷指令后台入账、预算超支提示、首次引导与全部异常路径（步骤 5–8）。

这样即便后台截图链路在真机上需要反复调，也不影响前两阶段先交付自用。

## 7. 架构行为设计

### 7.1 整体分层与数据流

```
                ┌──────────────────────── SwiftUI Views (4 Tab) ───────────────────────┐
                │  记账 · 账单 · 统计 · 我的  +  结果卡片 / 表单 / 弹窗                    │
                └───────────────▲───────────────────────────────────▲──────────────────┘
                                │ 绑定/调用                          │
                     ┌──────────┴───────────┐            ┌───────────┴───────────┐
                     │ @Observable ViewModel │            │ Analytics (聚合/趋势)  │
                     └──────────┬───────────┘            └───────────┬───────────┘
        ┌───────────────────────┼───────────────────────────────────┘
        │                       │
 ┌──────┴──────┐        ┌───────┴────────┐        ┌───────────────┐        ┌──────────────┐
 │ Capture 入口 │──文本──▶│ DeepSeek Client │──JSON──▶│  Transaction  │        │  Keychain    │
 │ 手/文/音/图  │        │ (URLSession)   │        │  写入 (M1)     │        │ (DeepSeek Key)│
 └──────┬──────┘        └────────────────┘        └───────┬───────┘        └──────────────┘
        │ 本地 OCR/Speech（图片/语音不外传）                 │
 ┌──────┴──────┐                                    ┌──────┴───────┐
 │ Vision/Speech│                                    │  SwiftData    │
 └─────────────┘                                    │ ModelContainer│
                                                     └──────────────┘
```

- 视图只做展示与交互，业务经 `@Observable` ViewModel/Store；数据经 SwiftData `ModelContext`。
- 识别类入口统一收敛到 **M4 DeepSeek Client** 的同一解析契约，四种入口只是"喂给它的文本从哪来"不同（手动不经 M4）。

### 7.2 识别类记账的统一状态机（文本/语音/相册截图，前台）

```
idle → capturing(取文本: 粘贴/Speech/OCR) → parsing(DeepSeek) →
   ├─ success → 直接入账(写 Transaction) → 弹结果卡片(可当场改) → done
   └─ fail/timeout/no-key → 保留原始文本 → 提示并转手动补录(原文带入)
```

- "识别中"为明确态并禁止重复提交；失败不丢原始文本。
- 结果卡片弹出时账单**已入账**，卡片是"看一眼 + 可改"，"删除这笔"= 撤销并二次确认。

### 7.3 截图·快捷指令后台入账链路（主入口，已确认走"后台全流程"）

```
用户在付款页触发快捷指令
        │  (系统把截图作为参数交给 App Intent)
        ▼
App Intent.perform()  ← 在后台执行（无前台 UI）
        │
        ├─ 1. 本地 Vision OCR 抽文本（图片不外传）
        ├─ 2. 读取 Keychain 里的 DeepSeek Key
        │        └─ 无 Key → 发"请先配置 Key"通知，结束（不记账）
        ├─ 3. DeepSeek 解析文本（URLSession，带超时）
        │        ├─ 成功且解析出金额 → 写 Transaction 入账 → 发"已记一笔"成功通知
        │        └─ 失败/超时/无网/无金额 → 不记账 → 保留原图 → 发"没识别出，点此补录"失败通知
        └─ 4. 结束；用户点通知：成功→结果卡片可改；失败→手动补录(原文/原图带入)
```

**关键技术约束与对策（必须在 M2.5 节点用真机验证）**：

- **后台执行时间预算**：App Intent 后台执行受系统时间限制，一次 DeepSeek 往返（尤其慢网）可能超时。对策：给 DeepSeek 调用设**明确超时**；超时即走失败通知分支，**绝不生成脏账、绝不悬挂**。
- **降级预案（方案 B 兜底）**：若真机实测后台预算**普遍不足**以完成一次网络往返，则该链路降级为——后台只做 **OCR + 保存原始文本 + 发通知**，DeepSeek 解析与入账改由用户**点通知进 App 后前台完成**。此降级只影响截图主入口的"是否全后台"，不影响其余三种入口。
- **存储访问**：App Intent 后台进程需能写入与主 App **同一** SwiftData 存储（`ModelContainer` 共享方式、是否需要 App Group）——列为 §11 首要展开项。
- **原图临时留存**：截图原图临时存本机，成功入账或用户放弃后清理，失败时保留供补录；具体存储位置/清理触发点在 M9 节点定。
- **真机依赖**：快捷指令触发、App Intents 后台执行、本地通知、后台联网在模拟器上不完整，此节点验收必须真机。
- **传图机制风险（与时间预算并列的头号验证项）**：快捷指令"如何把当前付款页图像可靠地交给 App Intent"（系统截图动作是否可用、分享面板传入的是图还是文本、是否需先截图再取最近照片）直接决定主入口能否成立，与后台时间预算同为 M2.5 头号真机验证项，建议先做一个最小 spike 验证形态可行再展开。

### 7.4 隐私与密钥

- Vision/Speech 全部本机执行；**只有识别出的文本**离开设备发往 DeepSeek。
- DeepSeek Key 仅存 Keychain，运行时读取；不落库、不入日志、不进源码。

### 7.5 统计聚合策略

- 数据量极小（十年几 MB），**v1 采用按需实时查询聚合**，不预建缓存表；若后续某档统计在真机上有明显卡顿，再在对应节点评估轻量缓存（列为可选，不在 v1 基线强制）。

## 8. 数据与状态设计

SwiftData `@Model`（字段为逻辑定义，具体类型/可空性/关系在 M1 节点 TRD 落地）：

### Transaction（账单）
- `id`: 唯一标识
- `amount`: **Decimal**（元，正值；方向单独表达）
- `direction`: 枚举 `expense | income`
- `occurredAt`: 发生时间（识别不到时取当前时间）
- `category`: 关系 → `Category`
- `merchant`: 商户/对方（可空）
- `note`: 备注（可空）
- `cardTail`: 卡尾号（可空，仅记录，不参与分账户统计）
- `source`: 枚举 `screenshotShortcut | screenshotAlbum | voice | sms/text | manual`（区分快捷指令 vs 相册仅作回溯）
- `rawText`: 原始输入文本（可空，供回溯/改正）
- `imageRef`: 截图临时引用（可空，仅截图来源，成功/放弃后清理）
- `createdAt` / `updatedAt`: 审计时间

### Category（分类）
- `id`、`name`、`direction`（收支属性）、`icon`/`color`（可选）、`isPreset`、`sortOrder`
- 预置：支出 衣/食/住/行/玩/其他；收入 工作/其他收入。归类约定：医疗/话费/数码等未单列支出 → "其他"；红包/退款/转账进账等未单列收入 → "其他收入"。用户可增删改。

### Budget（预算）
- `id`、`periodType`（枚举 `weekly | monthly`）、`amount`（Decimal）
- 周预算与月预算可同时存在（各一条）。

### BalanceBaseline（余额基线）
- `id`、`initialAmount`（Decimal）、`establishedAt`（录入时间点）
- **剩余金额（派生，不存储）** = `initialAmount` + Σ(基线后收入) − Σ(基线后支出)；无基线时视图显示"—"。

### 非实体状态
- **DeepSeek Key** → Keychain（不进 SwiftData）。
- **通知开关 / 预算周期规则 / 超支阈值(默认 80%)** → UserDefaults。

### 关键状态规则
- 周期：周按自然周（周一–周日）、月按自然月；支出累计到对应周期算进度。
- 剩余金额与统计在任意账单增删改后**同步刷新**（SwiftData 变更驱动视图）。

## 9. 接口与任务边界

### 9.1 DeepSeek 解析接口（M4 对外契约）
- **协议**：DeepSeek 的 OpenAI 兼容 Chat Completions HTTP 接口，`URLSession` 调用，Key 来自 Keychain。
- **输入**：原始文本 + 当前分类清单 + 输出格式约束（要求返回结构化 JSON）。
- **输出（解析目标字段）**：`amount`、`direction`、`occurredAt`、`merchant`、`cardTail`、`categoryName`。
- **归一/兜底**：金额转 Decimal（元）；无时间取当前；分类不匹配 → "其他/其他收入"；解析不出金额 → 判定为失败（由入口层决定转手动或发失败通知）。
- **错误类型**：无 Key / 网络失败 / 超时 / 无金额 / 非法响应——须可区分，供 §7.2、§7.3 分支。
- 具体 endpoint、模型名、prompt 文案、JSON schema、超时值、重试策略 → M4 节点 TRD。

### 9.2 本地识别接口边界
- **Vision**：输入图片数据 → 输出识别文本；仅本机。
- **Speech**：输入录音 → 输出转写文本；仅本机；需麦克风+语音识别权限。

### 9.3 App Intent（快捷指令）接口边界
- 暴露一个"记录 Aubade 截图"动作，参数为图片文件；`perform()` 后台执行 §7.3 链路。
- 与快捷指令的具体接入形态（App Intents in-app vs 扩展、传图方式、是否 App Group）→ M2.5 / §11。

### 9.4 通知接口边界
- 本地通知（成功/失败/无 Key）携带 `transactionId` 或"补录"意图，供点击路由；权限申请与开关在 M8。

### 9.5 任务边界总结
- **本基线定**：分层、模块边界、数据模型、集成契约的"输入/输出/错误类型"、开发顺序、验收口径。
- **节点 TRD 定**：文件级结构、具体类型与关系、prompt/JSON schema、超时/重试数值、容器共享的具体实现、UI 组件拆分。

## 10. 测试与验收口径

对齐 PRD 的 13 条验收点，测试分三层：

### 单元测试（纯逻辑，可脱离设备）

- 剩余金额计算（基线 + 收入 − 支出）。

- 周期边界（自然周周一起、自然月）。
- 为可测，DeepSeek/Vision/Speech 以**协议抽象 + mock** 注入。

### 集成 / UI 测试
- 四种入口各自入账成功（对应验收点 1–5）。
- 自动分类可改、改后统计同步（验收点 6）。
- 统计页周/月合计与分类占比正确（验收点 7）。

- 通知点击路由（成功→结果卡片、失败→补录）。
- 相册/麦克风/语音/通知权限被拒时的降级提示。

## 11. 节点 TRD 必须继续展开的问题

1. **App Intent 后台存储共享**（最高优先）：`ModelContainer` 如何被后台 Intent 进程与主 App 共享（in-app App Intents 直接共享，还是需 App Group + 共享容器）；后台写入的并发/一致性。**建库时机耦合**：若倾向"独立扩展进程"路线，则 M1 建库时就应按 App Group 共享容器建，避免后期存储迁移返工；in-app App Intents 路线则 M1 无需 App Group。此路线判定应尽量前移，不要拖到 M2.5。
2. **后台时间预算与降级判定**：真机实测后台预算是否够一次 DeepSeek 往返；不够时 §7.3 方案 B 降级的具体实现与切换标准。
3. **DeepSeek 调用细节**：endpoint、模型名、prompt 设计、返回 JSON schema、超时值、重试与限流处理。
4. **原图临时留存**：存储位置、命名、成功/放弃/失败三态下的清理触发点与时机。
5. **Keychain 封装**：读写接口、Key 缺失/变更处理、"已配置"状态判定。
6. **统计聚合实现**：四档区间与趋势序列的具体查询；是否需要针对某档做轻量缓存（默认不做，视真机表现定）。
7. **分类推断兜底**：DeepSeek 分类名 → 本地 Category 的匹配与"其他/其他收入"兜底的具体规则。
8. **首次引导流程**：录初始总额 + 提示配置 Key 的具体步骤、可跳过后的空账本落地。
9. **权限申请时机与被拒降级**：相册、麦克风+语音识别、通知的申请点与拒绝后的 UI 退路。
10. **快捷指令设置指引**：给用户的两步配置说明与 App 内说明卡文案。
