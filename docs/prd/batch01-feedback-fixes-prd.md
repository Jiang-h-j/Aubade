## 给用户看的摘要

你真机试用后提了 6 条，我核实了代码，性质分三类，这批一起做：

- **① 剩余总额记账后不变【真 bug】**：现在的算法只统计「你设初始总额那一刻之后」的账，早于那个时刻的账（尤其"先记账、后录初始总额"）被整个漏掉了。**按你定的口径修**：初始总额 = 你录入那一刻手上所有钱的净值，之后**每一笔**收支都在它上面加减，不再按日期卡。
- **② 账单顺序不对【一半是感知偏差、一半要修】**：账单列表代码其实已经按「消费日期」倒序排了。你觉得不对，是因为**识别入账时如果 AI 没解析出日期，就默认用了记账当下的时间**——所以那几笔的"消费日期"其实是记账时间。这批会让识别尽量取到真实消费日期，取不到时也说清楚。
- **③ 最近记录不能删【交互补齐】**：记账页的"最近记录"现在只能点进去改、不能删（删除只在账单页有）。给它补上删除（侧滑 + 二次确认），和账单页一致。
- **④ 分类不能自定义【新功能】**：现在分类是固定预置的、只能看。这批让你能**自己增删改分类**（名字/图标/颜色）。另外你说的"AI 识别后自动分类"其实**已经在跑了**——识别出的分类会自动填上，只是目前只能落到预置分类里；加了自定义分类后，AI 也能落到你新建的分类上。
- **⑤ 截图只能识别一笔【新功能，较大】**：现在一张截图只出一笔账。这批做成**一张截图能识别多笔**（比如一张账单截图里有好几笔消费），每笔可单独确认/改/删。
- **⑥ 界面不好看、没按原型【UI 大改】**：原型是"晨曦暖白 + 精致卡片"那种质感（暖米白底、晨曦渐变大卡、珊瑚色支出/青绿色收入、圆角卡片、粗体层次），现在的界面基本是系统默认的灰白列表。这批**按原型把视觉重做一遍**。

因为这 6 条量级差别很大（有几行的 bug 修复，也有 UI 整体重做），后面会拆成多个开发节点、按优先级一个个来，不会一次性堆上去。这份 PRD 先把「做什么、做到什么算完成」定清楚，请你评审。

## 业务目标

1. **让核心数字可信**：剩余总额、账单顺序是记账 App 的信任基石，任何一笔对不上都会让用户怀疑整个账。优先修复 ①②，确保"记一笔、数字就对"。
2. **补齐基本的编辑闭环**：账单能删、分类能自定义，是记账 App 的基本盘。③④ 让用户真正"掌控自己的账本"，而不是只能用预设。
3. **提升识别效率**：⑤ 截图多笔让"一张账单截图 → 一次记多笔"成为可能，减少重复操作，是截图入口的核心价值放大。
4. **兑现设计质感**：⑥ 把已定稿的原型视觉真正落地，让 App 从"能用的系统默认样子"变成"有品牌质感的成品"。

## 需求列表

> 每条标注：性质、当前代码事实、目标行为、关键决策/待确认。

### R1 剩余总额口径修复【bug｜正确性｜高优先级】

- **当前事实**：`BalanceCalculator.remaining(transactions:baseline:)`（`Aubade/Features/Analytics/BalanceCalculator.swift:12-18`）用 `transactions.filter { $0.occurredAt >= baseline.establishedAt }` 过滤，只累加"基线建立时刻之后"的账。这是 N02 的既有设计（代码注释「PRD 已确认约定 2，同刻计入」）。
- **问题**：消费日期早于"设初始总额时刻"的账被整体排除。典型触发：**先记几笔账、后在我的页/引导里录初始总额**；或把某笔的消费时间选到过去。表现为"记了账、剩余总额不动"。
- **目标行为（口径已由用户拍板）**：**初始总额 = 录入那一刻所有账户的当前净值；剩余 = 初始总额 + 全部收入 − 全部支出，不按 `occurredAt` 与 `establishedAt` 的先后过滤。**
- **修法**：去掉 `BalanceCalculator.remaining` 里的日期过滤，改为对全部 `transactions` 求和。
- **决策**：本条**推翻 N02「约定 2：仅计基线后账单」**。经核查，生产代码里"基线后过滤"的**唯一点**就是 `BalanceCalculator.swift:14`；`StatisticsAggregator`/`AnalyticsTabView` 用的是**统计周期区间过滤**（`occurredAt >= period.start && < period.end`），不依赖 `establishedAt`，不受本条影响。技术基线阶段仍需确认受影响的单测（如 `StatisticsAggregatorTests` 中剩余相关用例）并更新。
- **口径边界（用户拍板，技术基线需写明）**：口径为"初始总额 = 录入那一刻的当前净值" + "早于录入时刻的账也参与加减"。二者叠加时，**若用户填写的净值本已扣除过某笔早期消费、而该消费又作为账单存在，会被双重扣减**。用户已确认此口径；使用建议是"录初始总额时填当前净值，且不要再补录初始总额之前的历史账"，此提示在引导/我的页录入处应有体现。
- **baseline 为 nil（未录初始总额）仍返回 nil / 显示"未设置—"**：这是另一条正常设计分支，不改。

### R2 账单/识别的消费日期填充【核查+改进｜中优先级】

- **当前事实**：账单列表已按消费日期倒序——`@Query(sort: \Transaction.occurredAt, order: .reverse)`（`Aubade/Features/Ledger/LedgerTabView.swift:13`），分组与组内也按 `occurredAt`。**排序逻辑本身正确，无需改。**
- **问题根因**：识别入账时 `RecognitionNormalizer.occurredAt(_ date:now:)`（`Aubade/Features/Recognition/Parsing/RecognitionNormalizer.swift:18-21`）在 `date == nil`（DeepSeek 没解析出日期）时兜底返回 `now`（记账时刻）。于是这些账单的"消费日期"实为记账时间，列表看起来像"按记账时间排"。
- **目标行为**：① 让 DeepSeek 解析尽量抽取账单/短信中的**真实消费日期**（prompt 与 decode 层面）；② 确实取不到日期时，兜底策略需明确（保持 now 还是留空提示用户补），并让用户在结果卡片/编辑时可见可改。
- **决策/待确认**：兜底到 `now` 是否可接受？还是取不到日期时在结果卡片上**高亮提示"日期未识别，请确认"**？倾向后者（让用户有感知）。此条与 ⑤ 截图多笔的解析改造相关，可能合并到同一识别改造节点。

### R3 记账页"最近记录"支持删除【交互补齐｜中优先级】

- **当前事实**：记账页最近记录区块（`Aubade/Features/Record/RecordTabView.swift` 最近记录 + `editSheet`，约 `:361-396`）只支持点进编辑，**未注入 `onDelete`**；删除入口集中在账单页 `LedgerTabView`（`swipeActions` + 二次确认，约 `:219-225`）。按 N01 切片划分有意为之。
- **目标行为**：记账页最近记录支持删除，**复用账单页同款交互**（侧滑删除 + 二次确认），删除后剩余总额/统计同步更新。
- **修法**：给最近记录行加 `swipeActions`，或给 `editSheet` 注入 `onDelete`，复用既有 `EditorActions.makeDelete` 与二次确认。小改。

### R4 自定义分类（增删改）+ 自动分类确认【新功能｜中优先级】

- **当前事实**：
  - 模型 `LedgerCategory`（`Aubade/Models/LedgerCategory.swift`）已有 `name/direction/icon/color/isPreset/sortOrder` 字段——**做自定义分类不需要改模型**。
  - `LedgerStore`（`Aubade/Store/LedgerStore.swift`）有 `createCategory(name:direction:...isPreset:sortOrder:)`（`:27`，已支持建非预置分类）、通用 `delete`（`:120`）；**缺 `updateCategory`（重命名/改图标色）、缺"预置分类不可删/不可改"保护**。
  - 我的页分类区块 `categorySection`/`categoryTags`（`RootTabView.swift` 约 `:270-296`）是只读 capsule 标签流，无增删改入口。
  - **自动分类已实现**：`RecognitionNormalizer.category(name:direction:in:)`（`RecognitionNormalizer.swift:26-33`）按 name+direction 精确匹配库分类、不中兜底"其他/其他收入"，`recognizeAndSave` 已把归一后的分类写进账单。局限：只能命中库中已有分类。
- **目标行为**：
  1. 我的页分类区支持**新增/编辑/删除自定义分类**（名字、图标、颜色、所属方向支出/收入）。
  2. **预置分类保护**：预置分类不可删除、名称不可改（是否允许改图标/色待确认）；用户自定义分类可自由增删改。
  3. **自动分类自然增强**：加了自定义分类后，DeepSeek 识别出的分类名只要匹配到（含自定义），就自动归类——无需额外改识别逻辑（`category(...)` 已按库匹配）。
- **决策/待确认**：① 删除已被账单引用的分类时如何处理（禁止删 / 删后账单转"其他" / 提示）？② 是否让 DeepSeek 的分类清单 prompt 带上用户自定义分类（让 AI 更可能命中自定义分类）？

### R5 截图多笔识别【新功能｜较大｜中低优先级】

- **当前事实**：全链路单笔——解析契约 `ParsedTransaction` 是单结构体（`Aubade/Features/Recognition/Parsing/TransactionParsing.swift:5`）；DeepSeek prompt 明写"提取**一笔**账单"（`DeepSeekClient.swift:48`）、`decode` 只取 `choices[0]` 单个 JSON 对象（`:68-85`）；`BackgroundIntakeService.intake` 一张图落一笔（`BackgroundIntakeService.swift:43-53`）。
- **目标行为**：一张截图可识别**多笔**账单（如一张含多笔消费的账单截图），每笔在结果页可**单独查看/改/删/确认入账**。
- **修法量级（中~大）**：改解析契约（`ParsedTransaction` → 数组 / 新增多笔结构）、prompt（改为"提取所有账单"）、decode（解析数组）、入账编排（循环落多笔）、结果卡片 UI（单笔 → 多笔列表可逐条操作）。**不动 `Transaction` 数据模型。**
- **决策/待确认**：① 多笔识别覆盖哪些入口——只截图，还是文本识别也一并支持多笔？② 前台截图（相册选图）和后台截图（快捷指令 N06）是否都要多笔？后台多笔的通知如何呈现（"已记 3 笔"）？③ 部分笔识别失败时如何处理（成功的入账、失败的提示）？

### R6 UI 按原型还原【UI 大改｜视觉层｜低优先级但影响面广】

- **当前事实**：原型 `prototype/app/`（`styles.css` 定义"晨曦暖白·精致卡片"风：`--dawn` 晨曦渐变、`#f6f3ee` 暖米白底、珊瑚支出/青绿收入语义色、22px 大圆角、800 粗体层次；`app.js` 定义各页布局）。当前 SwiftUI 实现基本是**系统 List/Form 默认观感**（系统冷灰白、默认圆角字重）。
- **突出差距**（子 agent 核实）：
  1. **晨曦渐变 hero 缺失**：原型账单/我的顶部汇总是暖橙粉→淡蓝渐变大卡，SwiftUI 用系统灰底 `RoundedRectangle` / 纯 Section。
  2. **暖米白主题缺失**：原型 `#f6f3ee` 暖背景 + 语义色，SwiftUI 走系统冷灰白。
  3. **记账入口卡简化**：原型每格有彩色圆角图标底 + 副标题，SwiftUI `EntryButton` 仅 emoji+标题裸方块。
  4. **列表用系统 List 而非圆角卡片**：原型是圆角卡 + 极淡分隔。
  5. **字体层次弱化**：原型大量粗体 + 大圆角，SwiftUI 沿用系统默认。
- **目标行为**：建立一套设计 token（颜色/圆角/字重/间距/语义色），按原型还原关键页面（记账、账单、统计、我的、识别结果卡）的视觉：晨曦渐变 hero、暖米白主题、彩色入口卡、圆角卡片列表、粗体层次。
- **修法量级（大，纯视觉层）**：抽设计 token + 逐页改样式，**不动数据模型、不动业务逻辑**。
- **决策/待确认**：① 是否需要暗色模式适配（原型是否有暗色）？② 还原范围——全部页面一次到位，还是先核心页（记账+账单+我的）？③ 用户对原型的忠实度要求（像素级 vs 神似）？

## 优先级与阶段边界

| 优先级 | 需求 | 理由 | 量级 |
|---|---|---|---|
| **P0 正确性** | R1 剩余总额 | 核心数字可信，几行改动、立竿见影 | 小 |
| **P0 正确性** | R2 消费日期 | 与信任相关；核查为主 + 识别改进 | 小~中 |
| **P1 编辑闭环** | R3 最近记录删除 | 基本盘补齐，小改 | 小 |
| **P1 编辑闭环** | R4 自定义分类 | 基本盘，模型已够、补 Store API + UI | 中 |
| **P2 效率** | R5 截图多笔 | 价值放大，但改动大、决策多 | 中~大 |
| **P2 质感** | R6 UI 还原 | 影响面广、纯视觉、可最后统一做 | 大 |

- **阶段边界**：建议 DAG 阶段把 R1/R2（正确性修复，可能合成一个"核心数字修复"节点）→ R3/R4（编辑闭环）→ R5（识别多笔）→ R6（UI 还原）拆成有序节点。R6 放最后，避免视觉大改与逻辑改动交叉冲突。
- R2 与 R5 都涉及识别解析链路（`DeepSeekClient`/prompt/decode），DAG 阶段考虑是否合并为同一识别改造节点，减少对解析层的重复改动。

- **不在本批次动通知/权限/引导（N07 已收尾）**，除非 R1~R6 直接波及。

- **不重写导航架构**（仍是四 Tab）。

## 验收标准

> 每条给出可在真机/模拟器观察或单测验证的口径。

1. **R1**：先记 3 笔支出（各 10 元）、再录初始总额 1000 元 → 剩余显示 **970 元**（而非 1000）；再记 1 笔 20 元收入 → 剩余 **990 元**。把某笔消费日期改到初始总额之前 → 仍计入剩余。（单测：`BalanceCalculator.remaining` 覆盖"账单早于/晚于基线均计入"。）
2. **R2**：识别一条带明确日期的短信（如"6月3日消费58元"）→ 账单 `occurredAt` = 6月3日、列表排到对应日期位置；识别一条无日期文本 → 结果卡片明确提示"日期未识别/默认今天"，用户可改。（排序本身已对，验收落在日期填充与可见性。）
3. **R3**：记账页最近记录侧滑 → 出现删除 → 二次确认 → 删除后该笔消失、剩余总额与统计同步更新；与账单页删除行为一致。
4. **R4**：我的页新增一个自定义分类"宠物"（支出，自选图标色）→ 记账可选它；编辑重命名/删除自定义分类生效；预置分类不可删/不可改名（有保护提示）；识别一条"给猫买粮80元"且分类清单含"宠物" → 自动归到"宠物"（若 prompt 带自定义分类）。
5. **R5**：选一张含 3 笔消费的截图 → 结果页列出 3 笔，每笔可单独改/删 → 确认后账单页出现 3 笔；部分笔识别失败时，成功的入账、失败的有提示、不产生脏账。
