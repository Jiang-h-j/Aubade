# Aubade —— 可行性分析与接口调研

> 调研日期：2026-07-07
> 对应需求见《需求原始描述.md》

## 一句话结论

- **闹钟功能：完全可行。** iOS 26 的新框架 **AlarmKit** 就是为这类需求设计的。
- **记账功能：最难。** 微信和支付宝**都没有面向"个人付款方"的开发者接口**，能拿到的官方数据只有商户侧接口（收款方）和手动导出的账单文件；实时自动记账在 iOS 上只能靠"快捷指令 + 短信/截图"的曲线方案。

---

## 一、闹钟功能

**重要前提纠正**：第三方 App **无法读取或操作 iOS 自带时钟 App 里的闹钟**（那是封闭的）。但 iOS 26（2025 年底随 iOS 26 发布）新增的 **AlarmKit** 框架，让你自己的 App 能创建**与系统时钟同等权限**的闹钟。

| 能力 | 是否满足需求 |
|---|---|
| 创建多个系统级闹钟 | 可行。支持固定时间 + 重复（如每周一三五 7:00） |
| 绕过静音/专注模式响铃 | 可行。与 Apple 时钟同优先级 |
| 锁屏、灵动岛、Apple Watch 显示 | 可行。框架自带，无需自己画 UI |
| "关一个，其余自动全关" | 核心风险点，见下 |

### "关一个关全部"的实现思路

App 自己维护"这是同一组保险闹钟"的关系。用户在 App 内关掉一个 → 调用 `cancel` 取消这组里的其他闹钟。AlarmManager 还提供 `alarmUpdates` 事件流，可监听闹钟状态变化。

### 唯一需要实测验证的风险

如果用户是从**锁屏/灵动岛**（而非打开 App）关掉闹钟，App 能否在后台被可靠唤醒、从而联动取消其他闹钟。这取决于后台唤醒时机，是这个功能落地前必须先写 Demo 验证的一点。其余部分工程上很成熟（GitHub 已有多个完整示例）。

### 前提条件

- 需 iOS 26.0+。
- 需在 Info.plist 声明 `NSAlarmKitUsageDescription` 申请授权。
- 每个 App 的闹钟数量有系统上限（设几个保险闹钟够用）。

## 二、记账功能

### 2.1 微信/支付宝开发者接口调研（最关心的部分）

**核心结论：没有面向个人消费者的接口。** 支付宝开放平台和微信支付的**所有** API 都是给**商户/服务商（收款方）**用的：

| 接口类型 | 面向谁 | 个人付款方能用吗 |
|---|---|---|
| 支付结果通知 `notify_url` | 商户（别人付钱给他时推送） | 不能。你是付款方，收不到 |
| 对账单下载 API（支付宝 `bill.downloadurl.query` / 微信申请交易账单） | 商户下载自己的**收款**流水 | 不能。是收款账单，不是你的支出 |
| 分账、退款、支付分等 | 商户 | 不能 |

想接入这些接口，前提是有**营业执照**注册成商户。个人消费数据被明确排除在开放能力之外——这是微信/支付宝的产品策略，不是技术限制，短期不会变。

**个人消费数据的唯一官方途径 = 手动导出账单：**

- **支付宝**：我的 → 账单 → 右上角"…" → 开具交易流水证明 → **用于个人对账** → 发邮箱 → 下载加密 zip → CSV
- **微信**：我 → 服务 → 钱包 → 账单 → 下载账单 → **用于个人对账** → 发邮箱 → 下载加密 zip（2025 年 7 月起变 xlsx，密码在"微信支付"服务消息里）
- 特点：**T+1、批量、手动**，不是实时推送。

### 2.2 三种支付方式逐个评估

| 支付方式 | 实时自动记账 | 可行方案 |
|---|---|---|
| **银行卡（短信）** | 部分可行 | iOS **禁止** App 内直接读短信。但可用**快捷指令自动化**："收到银行短信"触发 → 正则提取金额/商户 → 通过 App 的 URL Scheme 回传。这是国内成熟做法（iOS 17+ 短信触发已可免二次确认） |
| **支付宝** | 很难 | 无接口、无短信。实时方案只能靠**截图 OCR**：支付后截图（或轻点背面）→ 快捷指令 OCR → 回传 App。或走 2.1 的月度 CSV 导入 |
| **微信** | 很难 | 同支付宝 |

**注意一个网上的错误说法**：有文章宣称"iOS 17 有通知内容提取 API，能读微信/支付宝支付通知自动记账"。**这个说法不准确**——快捷指令的自动化触发器里并没有"收到任意 App 通知"这一项，主流实时方案实际是截图 OCR。这点建议以真机实测为准，别按那个思路设计。

**关于 iOS 短信过滤扩展（`ILMessageFilterExtension`）**：它存在，但沙盒极严——只能处理**陌生号码**短信、**不能联网**、**不能写共享容器**，设计目的是拦垃圾短信，**无法**用来做记账数据采集。

## 三、落地建议

### MVP 优先级（按投入产出比）

1. **先做闹钟**——需求清晰、AlarmKit 现成、体验最好，先写 Demo 验证"锁屏关一个能否联动关其余"。
2. **记账做成"半自动 + 导入"两条腿：**
   - App 提供 **URL Scheme**（如 `aubade://expense?amount=..&account=..`），配合银行短信的快捷指令自动化 → 银行卡这条链路能做到近乎全自动。
   - App 提供 **CSV/xlsx 账单导入**（解析支付宝/微信月度流水，做去重）→ 覆盖支付宝/微信。
   - 支付宝/微信的"实时"，接受"截图 OCR + 手动确认"的现实，或干脆只做月度导入。

### 待确认的点

- 是否会写 Swift / 用 Xcode？决定下一步是出技术方案还是也要考虑现成方案。
- 记账更在意"每一笔实时不漏"，还是"月底对账清楚就行"？直接决定要不要投入做快捷指令那套。
